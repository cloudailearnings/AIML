Encoding the Word "Apple" in a Tokenizer

1. High-Level Idea

Modern Large Language Models (LLMs) do not work directly with raw words. Instead, they convert text into tokens, and then each token is mapped to a number (token ID).

Conceptually, the pipeline looks like this:

Text Characters / Bytes Subword Tokens Token IDs

For a word like "Apple", a tokenizer might treat it as a single token: 

"Apple" → ["Apple"] → [23495]

In this explanation we will use the example ID 23495 just to illustrate the process. (Real models will have their own specific IDs.)

2. Step-by-Step Encoding of "Apple"

2.1 Character & Byte Representation

Internally, text is represented as bytes. For ASCII characters in UTF-8, each character maps to one byte:

Character ASCII Code Binary
A 65 01000001
p 112 01110000
p 112 01110000
l 108 01101100
e 101 01100101

So the raw byte sequence for "Apple" is: 

[65, 112, 112, 108, 101]

2.2 Byte-Pair Encoding (BPE) / Subword Merges

Tokenizers like GPT’s use Byte-Pair Encoding (BPE) or similar subword methods. The idea is:

  • Start with characters or bytes as initial tokens.
  • Look at a huge text corpus.
  • Repeatedly merge the most frequent adjacent pairs into new tokens.

An example (simplified) merge sequence could be:


A + p   → "Ap"
p + p   → "pp"
"pp" + l  → "ppl"
"ppl" + e → "pple"
"Ap" + "pple" → "Apple"

After all learned merges are applied, the tokenizer might treat "Apple" as a single token: 

"Apple" → ["Apple"]

In some tokenizers it could also be split like ["App", "le"] or ["▁Apple"].

2.3 Token → Token ID

Each token string has a corresponding numeric ID in the tokenizer’s vocabulary. For illustration:

Token Token ID (example)
"A" 2
"App" 23494
"Apple" 23495
"ple" 23496

So the final numerical encoding becomes:

"Apple" → ["Apple"] → [23495]

4. Extra Details About the Token ID 23495

4.1 1️⃣ What does 23495 actually mean?

Think of a tokenizer vocabulary as a big table that maps token texts to IDs:

ID Token text
0 <pad>
1 <bos>
2 "A"
3 "B"
... ...
23494 "App"
23495 "Apple"
23496 "ple"
... ...

Here, 23495 is just the index in this table. The number was assigned when the tokenizer was trained. The digit 2 at the front has no special meaning – it is simply part of the decimal number 23495.

So the lookup process is:

"Apple" → find the row where token text = "Apple" → read its ID → 23495

That’s all. There is no logic like “2 comes from A” or “3 comes from p”.

4.2 2️⃣ How are such numbers assigned?

When building a tokenizer, the process is usually:

  1. Start with a base set of tokens (bytes or characters).
  2. Analyze a huge text corpus and count which adjacent pairs appear most often.
  3. Merge the most frequent pairs into new tokens (Ap, App, Apple, and so on).
  4. Each time a new token is created, assign the next unused integer ID.

So IDs progress in a simple sequence:

0, 1, 2, 3, 4, 5, ... 23494, 23495, 23496, ...

The ID is not computed from the letters of "Apple". It is just “the 23,495-th token in the vocabulary”.

4.4 4️⃣ One more important note

In this explanation, 23495 is an example ID, not the true ID from a specific model.

Different models (GPT-4, Llama-3, Mistral, etc.) use different vocabularies, so "Apple" may have different IDs in each tokenizer.

If you specify a particular model (for example, GPT-4 or Llama-3-8B), the exact ID can be obtained using a small Python script with that model’s tokenizer.
