1. Vector Query with Semantic Search

1.1 Detailed Explanation

A vector query[1] is a query that is executed against a collection of high-dimensional vectors (embeddings) instead of plain text. In semantic search[2], both the user query and the documents are converted into numerical vectors using an embedding model[3]. The vector database[4] then finds which document vectors are closest to the query vector according to a similarity metric such as cosine similarity[5].

Instead of asking “Does this document contain the exact word ‘login’?”, semantic search asks “Is the meaning of this document similar to the meaning of the query about logging in?”. This allows the system to match “sign in”, “log on”, or “access my account” to the same set of documents.

User Query: "How do I sign in?" | v [Embedding Model][3] | v Query Vector (e.g., 768 numbers) | v [Vector DB Similarity Search][4][5] | v Top-k Document Vectors | v Most relevant documents returned to user

1.2 When, Where, and Why We Need Vector Queries

  • When? - Users search using natural language questions. - Exact keyword matching does not give good results. - There are many synonyms and paraphrases in the data.
  • Where? - Documentation and FAQ search. - Chatbots and virtual assistants. - Recommendation systems (e.g., “similar articles/products”). - Code search and log search.
  • Why? - Captures meaning instead of only exact words. - Handles spelling differences and synonyms gracefully. - Integrates beautifully with large language models in Retrieval-Augmented Generation (RAG)[6] workflows.

1.3 Python Example: Vector Query with Semantic Search

This example uses sentence-transformers with faiss.

# pip install sentence-transformers faiss-cpu

from sentence_transformers import SentenceTransformer
import faiss
import numpy as np

# 1. Load an embedding model (creates vector representations of text)[3]
model = SentenceTransformer("all-MiniLM-L6-v2")

# 2. Our tiny document collection
documents = [
    "You can reset your password from the login page.",
    "Contact support if you cannot access your account.",
    "We store all documents in a secure object storage bucket.",
    "Learn how to integrate our product with Python."
]

# 3. Encode documents into a 2D array of vectors
doc_embeddings = model.encode(documents, convert_to_numpy=True)

# 4. Build a FAISS index for similarity search
dimension = doc_embeddings.shape[1]
index = faiss.IndexFlatIP(dimension)          # IP = inner product (works with normalized vectors)[5]

# Normalize embeddings for cosine similarity via inner product
doc_norms = np.linalg.norm(doc_embeddings, axis=1, keepdims=True)
doc_embeddings_normalized = doc_embeddings / doc_norms

index.add(doc_embeddings_normalized)

# 5. Encode the user query and normalize
query = "How do I sign in to my account?"
query_embedding = model.encode([query], convert_to_numpy=True)
query_embedding = query_embedding / np.linalg.norm(query_embedding, axis=1, keepdims=True)

# 6. Perform vector query (top 3 most similar)
k = 3
scores, indices = index.search(query_embedding, k)

print("Query:", query)
print("\nTop results:")
for rank, idx in enumerate(indices[0]):
    print(f"{rank+1}. {documents[idx]} (score={scores[0][rank]:.4f})")

2. Vector Embedding Models

2.1 Detailed Explanation

A vector embedding model[3] is a machine learning model that converts text (or images, audio, etc.) into numerical vectors. These models are typically built on top of transformer architectures[7] and are trained so that texts with similar meanings get similar vectors.

Common categories of embedding models include:

  • Sentence Transformers: e.g., all-MiniLM-L6-v2, optimized for sentence-level similarity.
  • LLM-based Embedding APIs: e.g., OpenAI, Azure OpenAI, where you send text and receive vectors.
  • Legacy Models: Word2Vec, GloVe, fastText – mainly word-level embeddings, less used for modern semantic search.
Text Input | v [Tokenizer][8] | v [Transformer Layers][7] | v [Pooling Operation] (e.g., mean of token embeddings) | v Embedding Vector (e.g., 384 / 768 dimensions)[9]

2.2 When, Where, and Why We Need Embedding Models

  • When? - Building any system that needs to understand similarity between texts. - Creating semantic search, clustering, topic discovery, or recommendation features.
  • Where? - Knowledge bases, help centers, document management systems. - E-commerce (similar products, product discovery). - Social media analysis, log analysis, anomaly detection.
  • Why? - Embeddings act as a universal numeric language for machine-learning models. - They allow you to use the same vector representation for multiple tasks: search, clustering, classification, etc.

2.3 Python Example: Comparing Two Embedding Models

# pip install sentence-transformers

from sentence_transformers import SentenceTransformer, util

# Model 1: small and fast
model_small = SentenceTransformer("all-MiniLM-L6-v2")

# Model 2: slightly larger (example - you can choose another one)
model_large = SentenceTransformer("all-mpnet-base-v2")

sent1 = "Login failed because the password was incorrect."
sent2 = "Authentication error: wrong password."

emb1_small = model_small.encode(sent1, convert_to_tensor=True)
emb2_small = model_small.encode(sent2, convert_to_tensor=True)

emb1_large = model_large.encode(sent1, convert_to_tensor=True)
emb2_large = model_large.encode(sent2, convert_to_tensor=True)

# Use cosine similarity to compare sentence pairs
sim_small = util.cos_sim(emb1_small, emb2_small).item()
sim_large = util.cos_sim(emb1_large, emb2_large).item()

print("Small model similarity :", sim_small)
print("Large model similarity :", sim_large)

3. Data Preparation

3.1 Detailed Explanation

Data preparation[10] is the process of turning raw content into a structured form that is ready for embedding and storage in a vector database. It usually includes cleaning, normalization, splitting long documents into chunks, and attaching metadata.

For example, a long PDF user guide may be split into smaller sections (paragraphs) so that each section can be retrieved individually when someone asks a question.

Raw Data Sources (PDFs, HTML pages, tickets, blogs) | v [Cleaning & Normalization] | v [Chunking into Paragraphs][11] | v [Attach Metadata][12] | v Ready-for-Embedding Records

3.2 When, Where, and Why We Need Data Preparation

  • When? - Before building an index in a vector database. - When integrating data from multiple sources with different formats.
  • Where? - ETL / ELT pipelines, data engineering workflows. - Analytics and search platforms.
  • Why? - Clean, well-structured data produces higher quality embeddings. - Smaller chunks and good metadata improve relevance and explainability of search results.

3.3 Python Example: Data Preparation for a Vector DB

import re
from textwrap import wrap

def clean_text(text: str) -> str:
    # Lowercase
    text = text.lower()
    # Remove HTML tags
    text = re.sub(r"<.*?>", " ", text)
    # Remove multiple spaces
    text = re.sub(r"\s+", " ", text).strip()
    return text

def chunk_text(text: str, max_chars: int = 300) -> list[str]:
    # Simple character-based chunking[11]
    return wrap(text, max_chars)

raw_doc = """
<h1>Login Troubleshooting</h1>
If you cannot login to your account, please check that
your username and password are correct. If the problem
continues, try resetting your password or contacting support.
"""

cleaned = clean_text(raw_doc)
chunks = chunk_text(cleaned, max_chars=120)

records = []
for i, chunk in enumerate(chunks):
    record = {
        "id": f"login-guide-{i}",
        "text": chunk,
        "metadata": {
            "section": "login_troubleshooting",   # metadata[12]
            "chunk_index": i
        }
    }
    records.append(record)

for r in records:
    print(r)

4. Vector Embedding in Batches

4.1 Detailed Explanation

When the dataset is large, embedding all records one-by-one is slow and inefficient. Instead, we process them in batches[13]. A batch is a group of items sent together to the model.

Batch processing uses the parallelism of modern hardware (GPUs/CPUs) and reduces overhead from repeated model calls or API requests.

Records: [doc1, doc2, doc3, doc4, doc5, doc6, ...] Split into Batches (size = 3): Batch 1: [doc1, doc2, doc3] Batch 2: [doc4, doc5, doc6] ... Each Batch -> Embedding Model -> Vectors stored in Vector DB

4.2 When, Where, and Why We Need Batch Embedding

  • When? - Ingesting large datasets: thousands or millions of documents. - Calling external embedding APIs with rate limits.
  • Where? - Initial index building stage, or regular re-indexing. - ETL jobs scheduled daily or hourly.
  • Why? - Much faster than per-document embedding. - More memory-efficient and cost-effective. - Allows graceful handling of failures (retry batch instead of entire dataset).

4.3 Python Example: Batch Embedding and Upsert

# pip install sentence-transformers chromadb

from sentence_transformers import SentenceTransformer
import chromadb

model = SentenceTransformer("all-MiniLM-L6-v2")
client = chromadb.PersistentClient(path="./batch_vector_store")

collection = client.get_or_create_collection(name="articles")

documents = [f"Sample article number {i}" for i in range(1, 21)]

def batch(iterable, size):
    for i in range(0, len(iterable), size):
        yield iterable[i:i + size]

batch_size = 5
for batch_index, docs_batch in enumerate(batch(documents, batch_size)):
    print(f"Processing batch {batch_index+1} with {len(docs_batch)} documents")
    embeddings = model.encode(docs_batch, convert_to_numpy=True)  # batch embedding[13]

    ids = [f"doc-{batch_index*batch_size + i}" for i in range(len(docs_batch))]
    metadatas = [{"batch": batch_index} for _ in docs_batch]

    # Many vector DB clients support passing embeddings directly;
    # with chroma, embeddings can be handled internally using embedding_function,
    # here we show the structure.
    collection.upsert(ids=ids, documents=docs_batch, metadatas=metadatas)

print("Batch embedding & upsert complete.")

5. NER Helper Function

5.1 Detailed Explanation

Named Entity Recognition (NER)[14] is the task of detecting important entities in text, such as people, organizations, locations, products, and dates. A NER helper function is a small utility that uses an NER model to extract entities and then uses them to enrich metadata or improve search.

For example, in a support ticket system, NER can find product names, customer names, or city names so that queries like “issues with Product X in Mumbai” work better.

Text: "Order 123 was shipped from Chennai to Mumbai using BlueDart." | v [NER Model][14] | v Entities: ("Chennai", LOCATION) ("Mumbai", LOCATION) ("BlueDart", ORG) ("123", NUMBER / ORDER_ID) | v Attach entities as metadata -> better filters and search

5.2 When, Where, and Why We Need NER Helpers

  • When? - Text contains many concrete entities (cities, people, products). - You want to support filters like “city=Mumbai” or “product=VectorDB Pro”.
  • Where? - Customer support tickets, CRM notes, shipping logs, legal documents. - News article classification or alerting systems.
  • Why? - Adds structured information on top of unstructured text. - Allows entity-based search, analytics and dashboards. - Helps LLMs provide more precise answers in RAG systems.

5.3 Python Example: NER Helper with spaCy

# pip install spacy
# python -m spacy download en_core_web_sm

import spacy

nlp = spacy.load("en_core_web_sm")

def ner_helper(text: str):
    doc = nlp(text)
    entities = []
    for ent in doc.ents:
        entities.append({"text": ent.text, "label": ent.label_})
    return entities

sample = "Order 123 was shipped from Chennai to Mumbai using BlueDart."

print("Text:", sample)
print("Entities:")
for e in ner_helper(sample):
    print(f" - {e['text']} ({e['label']})")

The entities returned by ner_helper can be added into metadata before sending data to the vector database.

6. Metadata Processing

6.1 Detailed Explanation

Metadata[12] is “data about data”. In a vector search system, each document (or chunk) usually has fields such as:

  • id – unique identifier.
  • source – where the text came from (PDF, wiki, ticket system).
  • category – topic or type.
  • created_at – timestamp.
  • entities – extracted via NER helper.

Metadata processing means designing, cleaning and enriching these fields so that queries can filter and rank results more effectively.

Document Chunk | +-- text: "Password reset steps for Product X." +-- metadata: { "source": "help_center", "product": "Product X", "language": "en", "created_at": "2025-11-01", "entities": ["Product X"] } Vector DB stores: (embedding_vector, metadata)

6.2 When, Where, and Why We Need Metadata Processing

  • When? - You want filters like “only show results for Product X”. - You need to separate internal vs external documents. - You want ranking rules such as “prefer newer documents”.
  • Where? - Enterprise search, multi-tenant SaaS platforms. - Knowledge bases with multiple products or business units.
  • Why? - Metadata filters greatly improve result relevance. - They enable efficient routing (e.g., which department owns this doc?). - They help with access control (e.g., user can see only their company’s docs).

6.3 Python Example: Preparing Metadata and Filtering in a Vector DB

# Example with chromadb; other vector DBs have similar metadata filters

# pip install chromadb sentence-transformers

import chromadb
from chromadb.utils.embedding_functions import SentenceTransformerEmbeddingFunction

client = chromadb.PersistentClient(path="./metadata_store")

embedding_func = SentenceTransformerEmbeddingFunction("all-MiniLM-L6-v2")

collection = client.get_or_create_collection(
    name="kb_with_metadata",
    embedding_function=embedding_func
)

docs = [
    "Password reset steps for Product X.",
    "Installation guide for Product Y on Linux.",
    "Troubleshooting Product X database connection issues.",
]

metadatas = [
    {"product": "X", "category": "auth", "language": "en"},
    {"product": "Y", "category": "install", "language": "en"},
    {"product": "X", "category": "troubleshooting", "language": "en"},
]

ids = ["doc-x-1", "doc-y-1", "doc-x-2"]

# Upsert with metadata
collection.upsert(ids=ids, documents=docs, metadatas=metadatas)

# Query with metadata filter: only Product X
query = "How do I fix login issues?"
result = collection.query(
    query_texts=[query],
    n_results=5,
    where={"product": "X"}  # metadata filter[12]
)

print("Query:", query)
print("Filtered results (product X only):")
print(result)

7. Interview Questions & Answers

Q1. What is a vector query and how is it used in semantic search?

Answer: A vector query is a query represented as an embedding vector[1]. In semantic search, the user’s text query is converted into a vector using an embedding model[3] and compared with document vectors stored in a vector database[4] using a similarity measure such as cosine similarity[5].

Q2. How do vector embedding models differ from traditional keyword-based approaches?

Answer: Embedding models convert text into high-dimensional vectors that encode meaning rather than exact word forms. Traditional keyword search relies on matching exact tokens. Embeddings allow “sign in” and “log in” to be close in vector space, which is not guaranteed in keyword-based systems.

Q3. Why is data preparation important before building a vector-based search system?

Answer: Data preparation ensures that the text is clean, normalized and split into appropriate chunks[11]. This leads to better embeddings, more relevant search results, and efficient storage. It also provides a consistent structure for attaching metadata[12].

Q4. What are the advantages of embedding documents in batches?

Answer: Batch embedding[13] reduces the overhead of calling the model many times, uses hardware efficiently, decreases total processing time, and fits nicely into ETL-style processing pipelines. It is especially useful for large datasets.

Q5. How can a NER helper function improve a semantic search system?

Answer: A NER helper function uses Named Entity Recognition[14] to extract entities such as products, locations, or people from text. These entities can be stored as metadata, enabling powerful filters (e.g., product = X) and more precise recall for queries involving specific names.

Q6. Give an example of how metadata can be used during a query.

Answer: A user may search for “database connection issues” but only for “Product X”. The vector database can apply a metadata filter {"product": "X"} so that only embeddings whose metadata product is X are considered during similarity search.

Q7. What is cosine similarity and why is it commonly used in vector search?

Answer: Cosine similarity[5] measures the cosine of the angle between two vectors. It focuses on orientation rather than magnitude, which works well for embeddings where direction encodes semantic meaning. It is scale-invariant and easy to compute.

Q8. How would you choose an embedding model for a new project?

Answer: Important factors include language coverage, latency, model size, licensing, and performance on similarity tasks. A small, fast model (like all-MiniLM-L6-v2) might be enough for many internal tools, while public-facing products with high quality requirements may benefit from larger, more accurate models or hosted APIs.

Q9. What role does chunking play in vector-based document retrieval?

Answer: Chunking[11] splits long documents into smaller parts such as paragraphs or sections. Each chunk gets its own embedding and metadata. This allows the system to return only the most relevant part of a long document instead of the entire file, improving relevance and user experience.

Q10. How do metadata and embeddings complement each other?

Answer: Embeddings capture semantic similarity, while metadata captures structured attributes such as product, language or access level. Together, they enable powerful queries like “find documents similar to this question but only for product X, created in the last 30 days and in English.”

8. Technical Footnotes / Glossary

[1] Vector: An ordered list of numbers representing a point in a high-dimensional space. In semantic search, each vector represents a text segment.
[2] Semantic Search: A search technique that focuses on the meaning of the query and documents, rather than simple keyword matching.
[3] Embedding Model: A machine-learning model that maps text (or other data) to dense vectors so that semantically similar inputs are close in vector space.
[4] Vector Database: A database optimized for storing and querying large collections of vectors, supporting similarity search and metadata filters.
[5] Cosine Similarity: A measure of similarity between two vectors based on the cosine of the angle between them. Values range from -1 (opposite) to 1 (identical).
[6] Retrieval-Augmented Generation (RAG): A pattern where an LLM retrieves relevant documents from a database and then uses them as context to generate an answer.
[7] Transformer Architecture: A neural network design based on self-attention mechanisms, widely used in modern language models such as BERT and GPT.
[8] Tokenization: The process of splitting text into smaller units (tokens) such as words or subwords used as inputs to language models.
[9] Embedding Dimension: The length of the vector produced by an embedding model, e.g., 384 or 768. Higher dimensional spaces can express more nuanced relationships but require more memory and compute.
[10] Data Preparation: The workflow of cleaning, normalizing, structuring and enriching data before it is used by downstream systems such as models or databases.
[11] Chunking: The act of breaking a long document into smaller segments, often by paragraph or character length, so that each piece can be retrieved independently.
[12] Metadata: Descriptive information about a document or chunk, such as source, product, creation time, language or extracted entities.
[13] Batch Processing: Processing many items together in groups (batches) rather than one at a time to improve efficiency and throughput.
[14] Named Entity Recognition (NER): An NLP task that finds and classifies entities (people, organizations, locations, dates, etc.) within text.
