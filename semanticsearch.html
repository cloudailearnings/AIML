1. Vector DB – Semantic Search

1.1 What is Semantic Search in a Vector Database?

Traditional search engines match exact words or simple keywords. Semantic search tries to understand the meaning of a sentence and not only the exact words used. It does this by converting text into numerical vectors[1], usually called embeddings[2].

A vector database[4] stores these embeddings and can quickly find which stored vectors are closest to the query vector using a similarity measure such as cosine similarity[6].

Simple idea:
If two pieces of text have similar meanings, their vectors will be close to each other in the high-dimensional space. The vector database searches for the nearest vectors.

1.2 When, Where, and Why Do We Use Semantic Search?

  • When? When users ask questions in natural language, type long queries, or use synonyms and you still want the system to find relevant information.
  • Where? In chatbots, documentation search, code search, customer-support portals, product recommendation engines, or any Retrieval-Augmented Generation (RAG) system.
  • Why? - It understands meaning instead of only exact keywords. - It can match “doctor” with “physician” or “car” with “automobile”. - It works well even if the user does not know the exact text that exists in the database.

1.3 Conceptual Diagram of Semantic Search

User Query Text | v [Text Encoder / Embedding Model][2] | v Query Vector (e.g., 768 numbers) --------- | Document Vectors in Vector DB | [doc1] [doc2] [doc3] ... [docN] | v Similarity Search (e.g., cosine similarity[6]) | v Top-K Most Similar Documents Returned

1.4 Python Example: Simple Semantic Search with Sentence Transformers + FAISS

The following code demonstrates how to create embeddings for a few sentences, store them in an in-memory index, and then perform a semantic search. (This is a minimal teaching example.) 

# Install first (run in terminal, not in Python):
# pip install sentence-transformers faiss-cpu

from sentence_transformers import SentenceTransformer
import faiss
import numpy as np

# 1. Load a small embedding model
model = SentenceTransformer("all-MiniLM-L6-v2")  # produces embedding vectors[1][2]

# 2. Our small document collection
documents = [
    "Shiva is worshipped as Dakshinamurthy, the cosmic teacher.",
    "Semantic search uses embeddings instead of plain keywords.",
    "A vector database stores and searches high-dimensional vectors.",
    "Ganesha is known as the remover of obstacles."
]

# 3. Convert documents to embeddings (2D array: num_docs x dim)
doc_embeddings = model.encode(documents, convert_to_numpy=True)

# 4. Build a FAISS index
dimension = doc_embeddings.shape[1]      # e.g., 384
index = faiss.IndexFlatL2(dimension)     # L2 distance; cosine also possible[6]
index.add(doc_embeddings)                # store vectors in the index

# 5. Take a user query and embed it
query = "Who is the cosmic teacher?"
query_embedding = model.encode([query], convert_to_numpy=True)

# 6. Search in the index (k = top 2 results)
k = 2
distances, indices = index.search(query_embedding, k)

print("Query:", query)
print("\nTop results:")
for rank, idx in enumerate(indices[0]):
    print(f"{rank+1}. {documents[idx]} (distance={distances[0][rank]:.4f})")

2. Data Preprocessing

2.1 What is Data Preprocessing?

Data preprocessing[7] means preparing raw text so that it is clean, consistent and ready for embedding and storage. Real-world text is messy: it can contain HTML tags, extra spaces, emojis, spelling mistakes, or repeated content.

Good preprocessing improves the quality of embeddings and therefore the quality of semantic search results.

2.2 When, Where, and Why Do We Use Data Preprocessing?

  • When? Before creating embeddings, before indexing in a vector database, or whenever data is collected from many different sources.
  • Where? - Building a documentation search system - Preparing support tickets for analysis - Preparing product descriptions for recommendation engines
  • Why? - Removes noise and irrelevant parts (for example, HTML headers or boilerplate). - Normalizes the text (“USA” vs “U.S.A.” vs “United States”). - Reduces storage and speeds up search.

2.3 Typical Steps in Text Preprocessing

  • Lowercasing all text.
  • Removing HTML tags.
  • Removing URLs, special symbols, or emojis if not needed.
  • Normalizing whitespace.
  • Removing stop words[8] like “is”, “the”, “a” (optional, depends on your model).
  • Applying lemmatization[9] to reduce words to their base form.

2.4 Diagram: Raw Text to Clean Text

Raw Support Ticket: "HELLO!! <br> I can't LOGIN into my account!!! 😭 Visit http://example.com/help" | v (lowercase, remove html, urls, normalize spaces) Clean Text: "hello i cannot login into my account"

2.5 Python Example: Simple Text Preprocessing Pipeline

# Install for advanced NLP (optional):
# pip install spacy
# python -m spacy download en_core_web_sm

import re
import spacy

nlp = spacy.load("en_core_web_sm")

def clean_text(text: str) -> str:
    # 1. Lowercase
    text = text.lower()

    # 2. Remove HTML tags
    text = re.sub(r"<.*?>", " ", text)

    # 3. Remove URLs
    text = re.sub(r"http\S+|www\.\S+", " ", text)

    # 4. Keep only letters, numbers and basic punctuation
    text = re.sub(r"[^a-z0-9\s\.,!?]", " ", text)

    # 5. Normalize spaces
    text = re.sub(r"\s+", " ", text).strip()

    return text

def lemmatize_and_remove_stopwords(text: str) -> str:
    doc = nlp(text)
    tokens = []
    for token in doc:
        # Skip stopwords and punctuation
        if token.is_stop or token.is_punct:
            continue
        # Lemma is the base form (e.g., "running" -> "run")[9]
        tokens.append(token.lemma_)
    return " ".join(tokens)

raw_text = "HELLO!! <br> I can't LOGIN into my account!!! 😭 Visit http://example.com/help"

step1 = clean_text(raw_text)
step2 = lemmatize_and_remove_stopwords(step1)

print("Raw text   :", raw_text)
print("Clean text :", step1)
print("Final text :", step2)

After preprocessing, the final text can be passed to an embedding model and stored in the vector database.

3. Preparing for Upsert

3.1 What is Upsert?

Upsert[10] is a combination of update and insert. If a record (or document) already exists in the database (based on its identifier / primary key[11]), it is updated. If it does not exist, it is inserted.

Many vector databases provide an upsert operation so that you can safely send data without checking separately whether it is new or existing.

3.2 When, Where, and Why Do We Need Upsert?

  • When? - When documents are frequently edited or corrected. - When you regularly sync data from another system (for example, a CMS or ticketing tool).
  • Where? - Product catalog search where product descriptions change. - Knowledge bases where articles are updated. - Log or event storage where data is appended but sometimes corrected.
  • Why? - Simplifies code: one API call can handle both insert and update. - Keeps the vector database consistent with the source of truth. - Avoids duplicated entries.

3.3 What Does “Preparing for Upsert” Mean?

Before calling an upsert operation in a vector DB, we usually:

  • Ensure every document has a stable and unique ID[11].
  • Preprocess the text (Section 2) to a clean format.
  • Generate or update the embedding for the cleaned text.
  • Optionally define metadata fields (e.g., source, category, tags, language).
  • Package all this information into a batch[12] for efficient upsert.

3.4 Diagram: End-to-End Flow for Upsert into a Vector DB

Raw Documents | v [Data Preprocessing][7] | v Clean Text | v [Embedding Model][2] ---> Embedding Vectors | \ | \ Metadata (title, tags, url) v [Upsert into Vector DB][10] | v Vector Database Ready for Semantic Search

3.5 Python Example: Preparing and Upserting into a Vector DB (Chroma-style API)

This example shows a simple way to prepare documents and upsert them using a Chroma-like client. The same ideas apply to Qdrant, Pinecone, etc. 

# Example only: a simple upsert-style workflow using ChromaDB-like API
# pip install chromadb sentence-transformers

import chromadb
from chromadb.utils.embedding_functions import SentenceTransformerEmbeddingFunction

# 1. Create persistent client
client = chromadb.PersistentClient(path="./my_vector_store")

# 2. Define embedding function
embedding_func = SentenceTransformerEmbeddingFunction(
    model_name="all-MiniLM-L6-v2"
)

# 3. Get or create a collection
collection = client.get_or_create_collection(
    name="knowledge_base",
    embedding_function=embedding_func
)

# 4. Our updated documents coming from some external system
docs_to_upsert = [
    {
        "id": "doc-001",  # stable unique ID (primary key)[11]
        "text": "Updated description of semantic search in our product.",
        "metadata": {"source": "product_docs", "category": "search"}
    },
    {
        "id": "doc-002",
        "text": "New help article about logging in and resetting passwords.",
        "metadata": {"source": "help_center", "category": "authentication"}
    }
]

# 5. Simple preprocessing (you can reuse the preprocess functions from Section 2)
def basic_clean(text: str) -> str:
    return " ".join(text.split()).strip()

ids = []
documents = []
metadatas = []

for item in docs_to_upsert:
    ids.append(item["id"])
    documents.append(basic_clean(item["text"]))
    metadatas.append(item["metadata"])

# 6. Upsert style call:
# Many libraries name this 'upsert', some use 'add' with 'update' flags.
collection.upsert(
    ids=ids,
    documents=documents,
    metadatas=metadatas
)

print("Upsert completed for IDs:", ids)

The next time search runs, the vector database will use the latest embeddings for these documents and return updated results.

4. Interview Questions & Answers

Q1. What is a vector database and why do we use it for semantic search?

Answer: A vector database[4] stores high-dimensional vectors that represent embeddings of text, images or other data. It is used for semantic search[3] because similarity between vectors corresponds to similarity in meaning. This allows us to find relevant documents even when the query words do not exactly match the document words.

Q2. How is semantic search different from keyword search?

Answer: Keyword search looks for exact word matches or simple variations. Semantic search compares embeddings of the query and documents. It can match phrases like “doctor” with “physician” or “sign in” with “login” because the embeddings capture meaning, not only surface words.

Q3. What is an embedding and how do we obtain it in Python?

Answer: An embedding[2] is a numerical vector representation of text (often 384 or 768 numbers) such that similar texts have similar vectors. In Python, we can use a library such as sentence-transformers: 

from sentence_transformers import SentenceTransformer
model = SentenceTransformer("all-MiniLM-L6-v2")
embedding = model.encode("Example sentence")

Q4. Why is data preprocessing important before building a semantic search system?

Answer: Data preprocessing[7] removes noise, normalizes text and ensures that the information sent to the embedding model is consistent. This leads to better embeddings, smaller index size, faster search and more accurate search results.

Q5. Give some common preprocessing steps for text data.

Answer: Common steps include lowercasing, removing HTML tags, removing URLs, normalizing spaces, optionally removing stop words[8], and applying lemmatization[9] so that related word forms share the same base form.

Q6. What is cosine similarity and where is it used in vector databases?

Answer: Cosine similarity[6] measures how similar two vectors are by looking at the angle between them. It is widely used in vector databases to compare query embeddings with stored document embeddings and to rank results.

Q7. Explain the term “upsert” in the context of a vector database.

Answer: Upsert[10] means “update if present, insert if not present”. When we upsert a document with a specific ID, the vector database either updates the existing record and its embedding or adds a new record if no such ID exists.

Q8. Why do we need a stable ID for each document when preparing for upsert?

Answer: A stable ID acts as the primary key[11]. The database uses this ID to decide whether a new document should be inserted or an existing one should be updated. If IDs are not stable, we may accidentally create duplicates or lose track of which record represents which document.

Q9. How would you connect data preprocessing to the upsert process in a production pipeline?

Answer: A typical pipeline first preprocesses each document, then generates embeddings for the cleaned text, and finally sends both the embeddings and metadata to the vector database via an upsert operation. This guarantees that the index always contains fresh, clean and consistent representations of the documents.

Q10. In Python, how can you batch multiple documents for efficient upsert?

Answer: Most vector DB clients allow passing lists of IDs, documents and metadata. For example: 

ids = ["doc-1", "doc-2", "doc-3"]
docs = ["text 1", "text 2", "text 3"]
metas = [{"source": "a"}, {"source": "b"}, {"source": "c"}]

collection.upsert(ids=ids, documents=docs, metadatas=metas)

Batching like this is an example of batch processing[12] and is much faster than sending one document at a time.

5. Technical Footnotes / Glossary

[1] Vector: A list of numbers (e.g., [0.12, -0.45, 0.88, ...]) used to represent data in mathematics and machine learning. In semantic search, each text segment is mapped to a vector in a high-dimensional space.
[2] Embedding: A dense numerical representation of text, image or other data that captures semantic properties such as similarity of meaning. Embeddings are usually produced by neural network models such as transformer encoders.
[3] Semantic Search: A search method that focuses on meaning, context and relationships between words, not only on exact matching of keywords. It uses embeddings and similarity measures.
[4] Vector Database: A database optimized for storing and querying large collections of vectors. It supports fast similarity search, indexing and metadata filtering.
[5] Token: A basic unit of text used by a model, often corresponding to a word or subword. Tokenization is the process of splitting text into tokens.
[6] Cosine Similarity: A similarity measure between two vectors based on the cosine of the angle between them. Value ranges from -1 (opposite) to 1 (identical). Higher cosine similarity means stronger semantic similarity between texts.
[7] Data Preprocessing: The set of operations applied to raw data to make it clean, consistent and ready for modeling. For text, typical steps include cleaning, normalization and token transformation.
[8] Stop Words: Very common words (such as “the”, “is”, “and”) that often carry little specific meaning. Some pipelines remove them to reduce noise, although many modern embedding models can handle them directly.
[9] Lemmatization: The process of reducing words to their base or dictionary form (lemma). For example, “running”, “runs” and “ran” all map to “run”.
[10] Upsert: A database operation that combines “update” and “insert”. If a record with a given ID exists, it is updated; if not, a new record is inserted.
[11] Primary Key / ID: A unique identifier used to distinguish one record from another in a database. In vector databases, the ID links each embedding back to its original document.
[12] Batch Processing: Handling multiple items at once instead of one by one. In vector databases, sending batches of documents for embedding or upsert is more efficient and faster.
